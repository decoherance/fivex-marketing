**Step 2: OG meta check**
Parse `dist/index.html` and verify these meta tags are present with correct values:
- `og:title` — should be "FiveX Strong — Strength training, simplified"
- `og:description` — should be "The simplest way to run a 5x5 strength programme. Just pick a workout and lift."
- `og:image` — should be "https://fivexstrong.decoherance-interactive.com/og-image.png"

```bash
node -e "
  const html = require('fs').readFileSync('./dist/index.html', 'utf8');
  const tags = ['og:title', 'og:description', 'og:image'];
  tags.forEach(tag => {
    const match = html.match(new RegExp('property=\"' + tag + '\" content=\"([^\"]+)\"'));
    console.log(tag + ':', match ? match[1] : 'MISSING');
  });
"
```

Record results for the checklist.

**Step 3: Alt text audit**
Parse `dist/index.html` for all `<img>` tags. Verify none have `alt="placeholder"` or empty `alt=""` (unless the image is decorative inside an `aria-hidden="true"` parent).

```bash
node -e "
  const html = require('fs').readFileSync('./dist/index.html', 'utf8');
  const imgs = [...html.matchAll(/<img[^>]*>/g)].map(m => m[0]);
  imgs.forEach(img => {
    const alt = img.match(/alt=\"([^\"]*)\"/);
    const src = img.match(/src=\"([^\"]+)\"/);
    const altVal = alt ? alt[1] : 'MISSING';
    const isEmpty = altVal === '' || altVal === 'MISSING';
    const isPlaceholder = altVal.toLowerCase() === 'placeholder';
    const flag = (isEmpty || isPlaceholder) ? ' <- FAIL' : '';
    console.log('src:', src?.[1], '| alt:', JSON.stringify(altVal) + flag);
  });
"
```

If any failures found, fix the source .astro file immediately, rebuild, and re-check.

**Step 4: Link resolution check**
Check the three footer links. Use Node's built-in fetch:

```bash
node --input-type=module -e "
const links = [
  { name: 'Play Store', url: 'https://play.google.com/store/apps/details?id=PLACEHOLDER_APP_ID' },
  { name: 'Privacy Policy', url: 'https://fivexstrong.decoherance-interactive.com/fivex/privacy-policy' }
];
for (const link of links) {
  try {
    const res = await fetch(link.url, { method: 'HEAD', redirect: 'follow' });
    console.log(link.name + ':', res.status, res.ok ? 'OK' : 'FAIL');
  } catch(e) {
    console.log(link.name + ': ERROR -', e.message);
  }
}
console.log('Support Email: mailto link - verify opens mail client (manual)');
"
```

For the Play Store URL: it will 404 because it uses `PLACEHOLDER_APP_ID`. This is expected and acceptable per the locked decision. Flag it as a pre-launch TODO, NOT a blocking failure.

For the privacy policy URL: if it 404s, this is a hard gate issue. Options:
- (a) Create a minimal privacy policy page at `src/pages/fivex/privacy-policy.astro`
- (b) Update `privacyPolicyUrl` in config.ts to point to an external URL that resolves

Choose the simplest fix that unblocks the gate. If creating a page, make it a minimal placeholder directing users to contact the support email for privacy inquiries.

Extract scores from each run:
```bash
for i in 1 2 3; do
```

Take the median of each category across the 3 runs. Median of [a, b, c] sorted = middle value.

**Step 3: Mobile viewport validation**
Since this is a CLI environment without a real browser, validate mobile responsiveness through the built HTML:

1. Check for `overflow-hidden` on the hero section (prevents horizontal scroll from absolute-positioned backgrounds):
   ```bash
   grep "overflow-hidden" dist/index.html
   ```

2. Check that no element has fixed pixel widths wider than 360px:
   ```bash
   grep -oP 'w-\[\d+px\]' dist/index.html | sort -u
   ```
   Any width value > 340px (360 minus padding) is a potential issue.

3. Verify body has the mobile padding for sticky CTA:
   ```bash
   grep "pb-20" dist/index.html
   ```

4. Verify sticky CTA has `lg:hidden`:
   ```bash
   grep -o 'id="sticky-cta"[^>]*' dist/index.html
   ```

5. Check that primary text uses `text-base` (16px) or larger — not `text-sm` for main paragraphs:
   ```bash
   # Verify the hero value prop uses text-lg or text-base
   grep "simplest way" dist/index.html | head -1
   ```

6. Validate touch targets meet 44x44px minimum. Run the Lighthouse accessibility audit and check the `tap-targets` diagnostic:
   ```bash
   npx lighthouse http://localhost:4321 \
     --only-categories=accessibility \
     --output=json \
     --output-path=./lighthouse-tap-targets.json \
     --chrome-flags="--headless=new --no-sandbox"

   node -e "
     const r = JSON.parse(require('fs').readFileSync('./lighthouse-tap-targets.json'));
     const audit = r.audits['tap-targets'];
     if (audit) {
       console.log('Tap Targets:', audit.score === 1 ? 'PASS' : 'FAIL (score: ' + audit.score + ')');
       if (audit.details && audit.details.items) {
         audit.details.items.forEach(item => {
           console.log('  -', item.tapTarget?.snippet || item.node?.snippet, '| size:', item.size, '| overlapping:', item.overlappingTarget?.snippet || 'none');
         });
       }
     } else {
       console.log('Tap Targets audit not found in results');
     }
   "
   rm -f lighthouse-tap-targets.json
   ```

   If any interactive elements fail the tap-targets audit (smaller than 48x48dp which covers the 44x44px minimum), fix the source .astro files by adding min-h-[44px] min-w-[44px] or equivalent Tailwind sizing. Rebuild and re-check.

For a thorough mobile check at the four viewport widths (360, 375, 768, 1280) in both orientations, note in the checklist that this requires manual browser testing and provide instructions for the user to verify.

**Step 2: Verify all assets load on deployed URL**
Fetch the deployed HTML page and extract all image/asset URLs, then verify each returns HTTP 200:

```bash
node --input-type=module -e "
const baseUrl = 'https://fivexstrong.decoherance-interactive.com';
const html = await (await fetch(baseUrl)).text();

// Extract image sources
const imgSrcs = [...html.matchAll(/src=[\"']([^\"']+\.(png|jpg|jpeg|webp|svg|ico|gif|avif))[\"']/gi)]
  .map(m => m[1]);

// Extract CSS/JS sources
const assetSrcs = [...html.matchAll(/(?:href|src)=[\"']([^\"']+\.(css|js))[\"']/gi)]
  .map(m => m[1]);

const allUrls = [...new Set([...imgSrcs, ...assetSrcs])];

console.log('Checking', allUrls.length, 'assets on deployed URL...');
let failures = 0;
for (const url of allUrls) {
  const fullUrl = url.startsWith('http') ? url : baseUrl + (url.startsWith('/') ? '' : '/') + url;
  try {
    const res = await fetch(fullUrl, { method: 'HEAD', redirect: 'follow' });
    const status = res.ok ? 'OK' : 'FAIL (' + res.status + ')';
    if (!res.ok) failures++;
    console.log(status, '-', fullUrl);
  } catch(e) {
    failures++;
    console.log('ERROR -', fullUrl, '-', e.message);
  }
}
console.log('\\nResult:', failures === 0 ? 'ALL ASSETS OK' : failures + ' FAILED');
"
```

If any assets fail, investigate — could be a missing file in the build, a wrong path, or a deployment issue. Fix and redeploy if needed.
